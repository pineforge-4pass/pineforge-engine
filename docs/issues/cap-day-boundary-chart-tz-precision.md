# Engine Bug — `max_intraday_filled_orders` cap-day boundary precision in chart-TZ-aware mode

**Filed:** 2026-05-17
**Severity:** Moderate (1 corpus probe affected: `cap-max-intraday-filled-orders-isolate-01`)
**Introduced by:** engine commit `6b062fc` (2026-05-15) — "feat(engine): 5 bug fixes + chart-TZ + harness overrides + 5 ctests"
**Pre-existing relative to Pine v6 sprint:** YES (not introduced this sprint)

## Symptom

When validator runs probe `cap-max-intraday-filled-orders-isolate-01` against
TV's `tv_trades.csv` (1958 trades, Asia/Taipei chart):

- Engine produces **1971 trades** (count delta +0.66%, but in compare window 4%)
- Trade-by-trade match_pct ≈ **51.2%**
- entry price drift p90 ≈ 0%
- exit price drift p90 ≈ 0.79%
- pnl rel diff p90 ≈ 100% (a handful of trades have wildly wrong exit prices)
- Validator scores **degree=2 (weak)**

Bisect (see `linked-doodling-beacon.md` audit phase) confirms:
- Pre-`6b062fc`: degree=3 (moderate), match_pct 82.2%
- Post-`6b062fc` (current main): degree=2 (weak), match_pct 51.2%

## Root cause (hypothesis)

`6b062fc` introduced `_decompose_bar_time_chart_tz()` (see
`include/pineforge/engine.hpp:355-365`, `_intraday_cap_currently_latched`)
to make `max_intraday_filled_orders` use chart-tz-aware day rollover instead
of UTC-day rollover. When the validator passes `chart_timezone="Asia/Taipei"`
(the default for this probe's TV csv interpretation, since
`tv_trades_csv_tz=utc_plus_8` → engine chart_tz="Asia/Taipei"), the cap-day
boundary fires at Asia/Taipei midnight (= 16:00 UTC).

The 4% extra trades and 49% mismatched-trade rate suggest the engine
*either*:
1. Computes Asia/Taipei midnight slightly wrong (e.g., off-by-one bar at
   the boundary), or
2. Triggers cap-close at a different bar than TV does (TV's broker
   emulator fires cap-close on the cap-triggering fill's bar; engine
   may fire on next bar or carry into the new day).

The stored `engine_trades.csv` (committed at probe creation time) shows
1958 trades 100% aligned to TV — but that csv was generated by
`run_strategy.py` with `--chart-tz ""` (UTC fast path), bypassing the
`6b062fc` chart-tz-aware code path. Hence the stored csv proves the
non-chart-tz path is correct.

## Failed remediation attempts

- **Setting `tv_trades_csv_tz=utc` in probe inputs.json:** repaired count
  delta (1957 vs 1955, 0.05%) but introduced systematic 8-hour timestamp
  display offset → match_pct dropped to **1.3%** (net worse). Reverted at
  corpus commit `30fb66c`.

## Investigation directions

1. **Diff cap-close trade-by-trade:** dump TV's 1958 trades and engine's
   1971 trades side-by-side (sorted by entry timestamp); identify which 13
   are extras and where the 49% mismatched ones diverge. Likely cluster
   around chart-day boundaries.
2. **Inspect `_decompose_bar_time_chart_tz()` boundary edge case:** does
   `dayofmonth * 100 + month` key match TV's chart-day partitioning at
   month-rollover days? E.g., 2025-04-30 23:55 Asia/Taipei vs 2025-05-01
   00:00 — does engine emit cap-close on the right side?
3. **Compare against pre-`6b062fc` UTC-only path:** add a probe-level
   override to force `chart_tz=""` regardless of csv_tz, run validator,
   confirm 100% match. This would prove `_decompose_bar_time_chart_tz()`
   is the sole bug locus.
